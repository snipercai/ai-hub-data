# 自适应问题导航器

## 📋 提示词内容

```
复杂问题:[描述问题]。不要立即回答。首先评估：你应该在这个问题上思考多深 (快速/中等/深度推理)、缺少什么信息、需要质疑什么假设。然后在适当的深度解决它。
```

## 🎯 适用场景

- 面临复杂的战略决策
- 需要深入分析的商业问题
- 技术方案选型和架构设计
- 风险评估和问题诊断
- 需要批判性思维的复杂场景

## 💡 使用说明

### 步骤 1: 描述问题

清晰描述你面临的复杂问题，包括背景、目标和已知约束。

### 步骤 2: 评估思考深度

AI 会判断问题需要的思考深度：
- **快速推理**: 简单问题，基于已有知识快速给出答案
- **中等推理**: 需要一定分析，考虑多个因素
- **深度推理**: 复杂问题，需要系统性分析和多角度思考

### 步骤 3: 识别缺失信息

AI 会指出：
- 缺少的关键信息
- 需要澄清的模糊点
- 可能需要验证的数据

### 步骤 4: 质疑假设

AI 会挑战：
- 你可能忽略的前提
- 未经验证的假设
- 潜在的认知偏见

### 步骤 5: 深度解决

基于以上分析，在适当的深度解决问题。

## 📝 使用示例

### 示例 1

**输入**:
```
复杂问题：我们应该将公司的单体应用迁移到微服务架构吗？目前团队 15 人，日活用户 10 万，系统偶尔出现性能问题。不要立即回答。首先评估思考深度、缺少信息和需要质疑的假设，然后解决它。
```

**输出**:
```
## 问题评估

### 思考深度：深度推理
这是战略性架构决策，影响深远，需要系统分析。

### 缺少的信息：
1. 当前性能问题的具体原因（架构问题还是资源问题？）
2. 团队对微服务的经验
3. 业务增长预期和规模规划
4. 开发和部署流程现状
5. 预算和时间约束

### 需要质疑的假设：
1. "微服务能解决性能问题" - 可能是数据库或缓存问题
2. "现在是迁移的好时机" - 可能业务快速增长期不是好时机
3. "团队能驾驭微服务复杂度" - 15 人团队可能偏小

## 深度分析

### 迁移到微服务的利弊：

**优势**：
- 独立扩展、技术栈灵活、部署解耦

**风险**：
- 运维复杂度增加、分布式事务、团队学习成本

### 建议方案：

**短期（1-3 个月）**：
1. 先优化当前瓶颈（数据库索引、缓存）
2. 模块化改造，为拆分做准备

**中期（3-6 个月）**：
1. 拆分 1-2 个独立服务试点
2. 建设 DevOps 能力

**长期**：
根据试点效果决定是否全面迁移

### 决策框架：
如果满足以下 3 个条件才考虑迁移：
- 性能问题确实由架构导致
- 团队有微服务经验或培训资源
- 业务稳定期，有 6-12 个月改造时间
```

### 示例 2

**输入**:
```
复杂问题：是否应该接受竞争对手的投资？我们是 A 轮创业公司，资金还能支撑 6 个月。不要立即回答，先评估再解决。
```

**输出**:
```
## 问题评估

### 思考深度：深度推理
这是关乎公司生存和发展的战略决策。

### 缺少的信息：
1. 投资金额和估值
2. 竞争对手投资的目的和期望
3. 其他融资渠道的可能性
4. 公司的盈利模式和增长情况
5. 投资条款（董事会席位、否决权等）

### 需要质疑的假设：
1. "只有接受投资才能活下去" - 可能有其他选择
2. "竞争对手是善意投资" - 可能是收购策略
3. "6 个月必须融资成功" - 可能通过盈利延长

## 分析框架

### 接受投资的风险：
- 战略自主权受限
- 敏感信息共享
- 未来退出受限
- 团队士气影响

### 可能的替代方案：
1. 加速盈利，减少融资依赖
2. 寻找战略投资者（非竞争）
3. 并购其他公司
4. 精简团队延长跑道

### 建议决策流程：
1. 评估所有融资选项
2. 咨询律师审查条款
3. 与团队核心成员讨论
4. 设定明确的底线和退出机制

### 如果必须接受：
- 争取最小化控制权条款
- 建立信息防火墙
- 设定明确的合作边界
- 准备 B 计划（回购条款等）
```

### 示例 3

**输入**:
```
复杂问题：产品上线后用户增长缓慢，应该加大营销投入还是改进产品？不要立即回答，先评估。
```

**输出**:
```
## 问题评估

### 思考深度：中等推理
需要分析产品和市场的匹配度。

### 缺少的信息：
1. 当前用户留存率和活跃度
2. 用户反馈和痛点
3. 与竞品的对比
4. 营销渠道和转化率
5. 产品核心指标（DAU/MAU、付费率等）

### 需要质疑的假设：
1. "问题要么在产品要么在营销" - 可能是市场定位问题
2. "加大投入就能解决" - 方向错了投入越多越糟

## 分析建议

### 先诊断问题根源：

**产品问题信号**：
- 留存率低（<30%）
- 用户活跃度下降
- 负面反馈多

**营销问题信号**：
- 留存好但新增少
- 目标用户认知度低
- 渠道覆盖不足

### 建议方案：

**第一步（2 周）**：深度用户调研
- 访谈 10-20 个用户
- 分析行为数据
- 找出核心问题

**第二步**：根据诊断结果决策
- 产品问题 → 改进产品，暂缓营销
- 营销问题 → 优化渠道，精准投放
- 混合问题 → 双线并行，优先产品

### 核心原则：
产品市场匹配（PMF）前，不要大规模营销
```

## 🔧 参数调整建议

- **问题复杂度**: 可根据问题性质调整评估标准
- **时间压力**: 紧急问题可缩短评估时间，但仍保持批判性思维
- **风险承受**: 高风险决策需要更深度分析
- **利益相关者**: 涉及多方的问题需要更全面的视角

## ⚠️ 注意事项

- 不要跳过评估阶段直接求解
- 勇于质疑自己的假设和偏见
- 承认信息不足时的不确定性
- 重大决策建议结合专家意见
- 定期回顾决策质量，改进思维模式
