# 跨库连接器

## 📋 提示词内容

```
我有两个项目:[项目 A] 和 [项目 B]。加载两个代码库。找出它们应该如何集成、需要创建哪些 API 端点、需要的共享数据结构、迁移策略和连接会破坏什么。绘制完整集成地图。
```

## 🎯 适用场景

- 并购后的系统整合
- 微服务之间的集成
- 遗留系统与现代系统集成
- 第三方系统对接
- 数据迁移和系统替换

## 💡 使用说明

### 步骤 1: 加载代码库

提供两个项目的相关信息：
- 上传关键代码文件
- 说明技术栈和架构
- 描述业务功能和数据模型

### 步骤 2: 分析集成点

AI 会分析：
- 功能重叠和互补
- 数据流和依赖关系
- 潜在的集成模式
- 技术兼容性

### 步骤 3: 设计 API 端点

确定需要创建的 API：
- 数据同步接口
- 业务调用接口
- 认证授权接口
- 事件通知接口

### 步骤 4: 定义共享数据结构

设计：
- 通用数据模型
- 消息格式
- 错误码规范
- 版本控制策略

### 步骤 5: 制定迁移策略

规划：
- 迁移阶段和步骤
- 数据迁移方案
- 回滚计划
- 风险控制

### 步骤 6: 识别破坏点

预警：
- 不兼容的变更
- 影响的功能模块
- 需要修改的代码
- 测试重点

## 📝 使用示例

### 示例 1

**输入**:
```
我有两个项目：电商订单系统（项目 A）和 CRM 客户系统（项目 B）。加载两个代码库。找出它们应该如何集成、需要创建哪些 API 端点、需要的共享数据结构、迁移策略和连接会破坏什么。绘制完整集成地图。

项目 A（订单系统）:
- 技术栈：Node.js + Express + MongoDB
- 核心功能：订单创建、支付处理、物流跟踪
- 数据模型：Order、OrderItem、Payment、Shipment

项目 B（CRM 系统）:
- 技术栈：Python + Django + PostgreSQL
- 核心功能：客户管理、销售管道、客服工单
- 数据模型：Customer、Contact、Deal、Ticket
```

**输出**:
```
## 跨库集成地图 - 订单系统 × CRM 系统

### 🗺️ 集成架构

```
┌─────────────┐         ┌─────────────┐
│  订单系统 A  │◄───────►│  CRM 系统 B  │
│  Node.js    │  REST   │  Python     │
│  MongoDB    │  API    │  PostgreSQL │
└─────────────┘         └─────────────┘
```

### 🔗 集成点分析

**1. 客户数据同步**
- 订单系统需要客户信息
- CRM 是客户数据源
- 集成方式：CRM 提供客户查询 API

**2. 订单数据同步**
- CRM 需要显示客户订单
- 订单系统是数据源
- 集成方式：订单系统提供订单查询 API

**3. 事件通知**
- 新订单 → 通知 CRM 创建销售机会
- 客户升级 → 通知订单系统更新权益

### 📡 需要创建的 API 端点

**订单系统提供**:
```
GET  /api/customers/:id/orders      # 获取客户订单
GET  /api/orders/:id                # 订单详情
POST /api/webhooks/crm              # CRM 回调通知
```

**CRM 系统提供**:
```
GET  /api/customers/:id             # 客户详情
PUT  /api/customers/:id             # 更新客户
POST /api/webhooks/orders           # 订单事件回调
```

### 📊 共享数据结构

**Customer（共享）**:
```json
{
  "id": "string",
  "email": "string",
  "name": "string",
  "phone": "string",
  "level": "bronze|silver|gold",
  "createdAt": "datetime",
  "updatedAt": "datetime"
}
```

**OrderSummary（订单系统→CRM）**:
```json
{
  "orderId": "string",
  "customerId": "string",
  "amount": "number",
  "status": "string",
  "createdAt": "datetime"
}
```

### 🚀 迁移策略

**阶段 1：准备（1 周）**
- 定义 API 规范
- 建立认证机制
- 准备测试环境

**阶段 2：单向同步（2 周）**
- CRM → 订单：客户数据同步
- 只读集成，风险低

**阶段 3：双向同步（2 周）**
- 订单 → CRM：订单数据同步
- 事件通知机制

**阶段 4：深度集成（2 周）**
- 业务流程打通
- 统一用户体验

### ⚠️ 破坏点预警

**订单系统需要修改**:
1. 添加客户数据缓存（减少 CRM 调用）
2. 订单创建流程增加 CRM 通知
3. 数据库增加 customer_id 索引

**CRM 系统需要修改**:
1. 客户详情页集成订单展示
2. 销售管道关联订单数据
3. 增加订单事件处理器

**潜在风险**:
1. 数据一致性问题（网络故障）
2. 性能影响（跨系统调用）
3. 认证授权复杂度增加

### ✅ 测试重点

1. API 兼容性和错误处理
2. 数据同步一致性
3. 网络故障容错
4. 性能压力测试
5. 安全认证测试
```

### 示例 2

**输入**:
```
我有两个项目：旧的用户管理系统（PHP + MySQL）和新的身份认证服务（Go + Redis）。需要集成它们，最终替换旧系统。请分析集成方案。
```

**输出**:
```
## 系统集成方案 - 用户管理迁移

### 🎯 集成策略：绞杀者模式

逐步用新系统替换旧系统，而非一次性迁移。

### 📡 API 端点设计

**新系统提供**:
```
POST /auth/login          # 登录认证
POST /auth/logout         # 登出
POST /auth/refresh        # 刷新令牌
GET  /users/:id           # 用户信息
POST /users               # 创建用户
PUT  /users/:id           # 更新用户
```

### 🔄 数据同步方案

**阶段 1：双写**
- 用户创建/更新同时写入两个系统
- 旧系统仍为读主数据源

**阶段 2：数据迁移**
- 批量迁移历史用户数据
- 增量同步变更数据

**阶段 3：切换读取**
- 逐步将读取流量切换到新系统
- 按用户 ID 哈希分流

**阶段 4：完全切换**
- 100% 流量到新系统
- 旧系统只读备份

### ⚠️ 破坏点

**需要修改的地方**:
1. 所有系统的用户认证调用
2. 用户数据查询接口
3. 会话管理机制

**兼容性处理**:
- 密码加密算法不同（MD5→bcrypt）
- 首次登录时迁移密码
- Session 令牌格式变更

### 📊 迁移检查清单

- [ ] API 网关配置
- [ ] 数据迁移脚本
- [ ] 回滚方案
- [ ] 监控告警
- [ ] 性能测试
- [ ] 安全审计
```

### 示例 3

**输入**:
```
我有两个微服务：支付服务和账务服务。需要集成它们实现交易处理。请设计集成方案。
```

**输出**:
```
## 支付服务 × 账务服务集成方案

### 🏗️ 集成架构

**同步调用**: 支付结果通知
**异步事件**: 交易记录同步

### 📡 API 设计

**支付服务→账务服务**:
```
POST /api/transactions          # 创建交易记录
PUT  /api/transactions/:id      # 更新交易状态
```

**账务服务→支付服务**:
```
GET  /api/payments/:id/status   # 查询支付状态
POST /api/refunds               # 发起退款
```

### 📨 事件设计

**PaymentCompleted**:
```json
{
  "eventId": "string",
  "type": "PaymentCompleted",
  "data": {
    "paymentId": "string",
    "amount": "number",
    "userId": "string",
    "timestamp": "datetime"
  }
}
```

### ⚠️ 一致性保障

1. 分布式事务：Saga 模式
2. 消息队列：确保事件不丢失
3. 对账机制：每日核对数据

### 🔒 幂等性设计

- 交易号唯一性约束
- 去重表记录已处理事件
- 重试机制和补偿逻辑
```

## 🔧 参数调整建议

- **集成深度**: 浅层（数据同步）/ 中层（API 调用）/ 深层（业务流程打通）
- **迁移速度**: 快速切换（风险高）/ 渐进式（风险低）
- **数据一致性**: 强一致性（性能低）/ 最终一致性（性能高）
- **监控粒度**: 基础监控 / 详细追踪 / 全链路监控

## ⚠️ 注意事项

- 充分测试后再上线
- 准备完善的回滚方案
- 监控关键指标
- 通知所有干系人
- 文档化集成细节
- 考虑安全和合规要求
