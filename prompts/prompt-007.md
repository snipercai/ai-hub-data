# 长期项目助手

## 📋 提示词内容

```
这是一个月长的项目:[描述项目]。追踪过去对话的决策、几周前的背景、不断变化的需求和我们已经尝试过的。使用上下文压缩，永远不要忘记任何重要内容。
```

## 🎯 适用场景

- 持续数周或数月的长期项目
- 涉及多轮对话和迭代的项目
- 需求和方案不断演化的项目
- 需要记住大量历史决策的项目
- 跨多个会话的复杂协作

## 💡 使用说明

### 步骤 1: 描述项目

提供项目的完整描述，包括：
- 项目目标和范围
- 时间周期
- 关键干系人
- 当前阶段

### 步骤 2: 建立历史追踪

AI 会记录和追踪：
- 每次对话的关键决策
- 方案演化的历史
- 尝试过的方法和结果
- 需求变化的时间线

### 步骤 3: 上下文压缩

使用压缩技术：
- 提炼关键信息
- 保留决策依据
- 记录失败教训
- 标记待办事项

### 步骤 4: 持续记忆

在长期对话中：
- 主动引用历史信息
- 提醒之前的决策
- 避免重复尝试错误
- 保持项目连贯性

## 📝 使用示例

### 示例 1

**输入**:
```
这是一个月长的项目：开发企业级 CRM 系统。项目从 3 月 1 日开始，预计 4 月 30 日完成。追踪过去对话的决策、几周前的背景、不断变化的需求和我们已经尝试过的。使用上下文压缩，永远不要忘记任何重要内容。

项目背景：
- 团队：8 人（2 前端、3 后端、2 测试、1PM）
- 技术栈：React + Spring Boot + MySQL
- 核心功能：客户管理、销售管道、报表分析

第 1 周决策：
- 选择微服务架构
- 使用 JWT 认证
- 集成企业微信

第 2 周变更：
- 新增移动端需求
- 报表功能优先级提升
- 截止日期推迟到 5 月 15 日

请总结你记住的内容。
```

**输出**:
```
## 长期项目记忆库 - CRM 系统开发

### 📊 项目概况
- **周期**: 3.1 - 5.15（因需求变更延期）
- **团队**: 8 人
- **技术栈**: React + Spring Boot + MySQL

### ✅ 关键决策（第 1 周）
1. 架构：微服务
2. 认证：JWT
3. 集成：企业微信

### 🔄 需求变更（第 2 周）
1. 新增：移动端适配
2. 优先级：报表功能提升
3. 时间：延期至 5.15

### 📝 已尝试方案
- 待补充后续对话内容

### ⏳ 待办事项
- 移动端技术选型
- 报表需求细化

### 💾 上下文压缩状态
- 保留：所有决策、变更、依赖
- 压缩：日常讨论细节
- 标记：关键里程碑和风险

我会在后续对话中持续追踪和引用这些信息。
```

### 示例 2

**输入**:
```
这是一个月长的项目：AI 客服系统优化。我们已经合作 3 周了，追踪所有历史。

第 1 周：
- 问题：响应速度慢，准确率 65%
- 尝试：优化模型参数，提升到 68%
- 决策：需要重新训练模型

第 2 周：
- 收集了 10000 条对话数据
- 尝试了 BERT 微调，准确率 75%
- 问题：推理速度变慢（2s→5s）

第 3 周（当前）：
- 需要平衡准确率和速度
- 考虑模型蒸馏
- 客户期望 80% 准确率

请追踪所有历史并给出建议。
```

**输出**:
```
## AI 客服系统优化 - 历史追踪

### 📈 进展时间线

**第 1 周（基线）**
- 问题：响应慢，准确率 65%
- 尝试：参数优化 → 68%（+3%）
- 结论：需要模型层面改进

**第 2 周（突破）**
- 数据：收集 10000 条对话
- 尝试：BERT 微调 → 75%（+7%）
- 新问题：推理 2s→5s（性能下降）

**第 3 周（当前）**
- 目标：准确率 80%，保持速度
- 考虑：模型蒸馏

### 💡 历史经验
1. 参数调优效果有限（+3%）
2. 大模型提升准确但牺牲速度
3. 数据质量是关键

### 🎯 建议方案

**模型蒸馏路线**：
- 教师模型：BERT（75% 准确率）
- 学生模型：轻量级（目标<1s）
- 预期：准确率 78-80%，速度<1s

**备选方案**：
1. 模型剪枝 + 量化
2. 缓存高频问题答案
3. 分层处理（简单问题规则引擎）

### ⚠️ 风险提示
- 蒸馏可能损失 2-5% 准确率
- 需要 1-2 周实验验证
- 建议与客户沟通期望值

### 📋 下一步
1. 搭建蒸馏实验环境
2. 准备训练数据
3. 设定评估指标
```

### 示例 3

**输入**:
```
这是一个月长的项目：电商平台重构。现在是第 4 周，请回顾所有历史。

前 3 周历史：
- 第 1 周：需求分析，确定微服务拆分方案
- 第 2 周：用户服务开发完成，订单服务开始
- 第 3 周：订单服务遇到分布式事务问题，尝试了 Saga 模式
- 当前问题：Saga 实现复杂，团队学习成本高

之前决策：
- 数据库按服务拆分
- 使用事件驱动
- 选择 Go 语言

已尝试方案：
- 2PC：性能太差，放弃
- Saga：复杂度高，进行中
- 本地消息表：待尝试

请总结并给出建议。
```

**输出**:
```
## 电商平台重构 - 第 4 周回顾

### 📅 项目历程

**第 1 周：架构设计**
- ✅ 需求分析完成
- ✅ 微服务拆分方案确定
- 决策：数据库拆分、事件驱动、Go 语言

**第 2 周：开发进展**
- ✅ 用户服务完成
- 🔄 订单服务启动

**第 3 周：技术挑战**
- 问题：分布式事务
- 尝试 1:2PC → 性能差，放弃
- 尝试 2: Saga → 复杂度高，进行中

**第 4 周：当前状态**
- 问题：Saga 学习成本高
- 待尝试：本地消息表

### 🔍 问题分析

**Saga 模式的挑战**：
- 优点：性能好，无锁
- 缺点：补偿逻辑复杂，调试困难
- 团队：缺乏相关经验

**本地消息表方案**：
- 优点：实现简单，易理解
- 缺点：性能一般，需要轮询
- 适用：订单量不大的场景

### 💡 建议

**方案 A：继续 Saga**
- 投入：1 周学习 + 实现
- 长期收益：性能好，可扩展
- 适合：高并发场景

**方案 B：切换本地消息表**
- 投入：3-5 天实现
- 快速上线，后续优化
- 适合：快速交付

**方案 C：混合方案**
- 核心业务：Saga
- 非核心：本地消息表

### 📊 决策建议
如果项目时间紧张 → 方案 B
如果追求长期性能 → 方案 A
如果资源允许 → 方案 C

### ⏳ 待办
- 确定事务方案
- 订单服务完成
- 支付服务启动
```

## 🔧 参数调整建议

- **记忆详细度**: 可根据项目复杂度调整（精简/标准/详细）
- **回顾频率**: 建议每周或每关键节点回顾
- **压缩策略**: 保留决策和结果，压缩讨论过程
- **提醒方式**: 主动提醒或被动查询

## ⚠️ 注意事项

- 定期（每周）更新项目状态
- 重大变更及时通知 AI
- 敏感信息需要脱敏
- 关键决策保留详细依据
- 定期导出重要信息备份
