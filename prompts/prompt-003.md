# 多文件项目审计员

## 📋 提示词内容

```
加载我的整个项目文件夹:[上传文件]。分析：需要改变的架构决策、首先要处理的技术债、安全漏洞、性能瓶颈和过时依赖。给我一个优先级行动计划。
```

## 🎯 适用场景

- 接手旧项目需要快速了解技术状况
- 准备进行大规模重构前的评估
- 定期代码质量审查
- 投资尽职调查中的技术评估
- 准备开源项目前的清理

## 💡 使用说明

### 步骤 1: 加载项目

上传整个项目文件夹或关键文件，确保包含：
- 源代码文件
- 配置文件（package.json, requirements.txt 等）
- 构建脚本
- 测试文件

### 步骤 2: 全面分析

审计员将从以下维度进行分析：
- 架构设计和模块组织
- 代码质量和可维护性
- 安全漏洞和风险点
- 性能瓶颈和优化空间
- 依赖库的版本和安全性

### 步骤 3: 生成报告

获取详细的分析报告，包括：
- 问题清单和严重程度
- 具体代码位置和示例
- 潜在影响和风险评估

### 步骤 4: 制定计划

根据优先级获得行动计划：
- 紧急问题（立即处理）
- 重要问题（短期处理）
- 改进建议（中期规划）
- 优化方向（长期目标）

## 📝 使用示例

### 示例 1

**输入**:
```
加载我的整个项目文件夹:[上传 web 应用项目]。分析：需要改变的架构决策、首先要处理的技术债、安全漏洞、性能瓶颈和过时依赖。给我一个优先级行动计划。
```

**输出**:
```
## 项目审计报告

### 🔴 紧急问题（立即处理）
1. **安全漏洞**: 用户输入未验证，存在 SQL 注入风险
   - 位置：/src/api/user.js 第 45 行
   - 影响：高危
   - 修复：使用参数化查询

2. **过时依赖**: lodash 4.17.15 存在已知漏洞
   - 升级到 4.17.21+

### 🟡 重要问题（本周处理）
1. **技术债务**: 重复代码率 35%
2. **性能瓶颈**: 数据库查询未使用索引

### 🟢 架构改进（本月规划）
1. 单体架构拆分为微服务
2. 引入缓存层

### 行动计划：
第 1 周：修复安全漏洞
第 2 周：更新依赖，重构重复代码
第 3-4 周：性能优化和架构调整
```

### 示例 2

**输入**:
```
加载我的整个项目文件夹:[上传 Python 数据分析项目]。进行全面审计并给出优先级行动计划。
```

**输出**:
```
## 数据分析项目审计

### 关键发现：

**架构问题**:
- 数据处理逻辑耦合严重
- 缺少模块化设计

**技术债务**:
- 函数平均长度 80 行（建议<30）
- 单元测试覆盖率仅 45%

**性能问题**:
- 使用循环而非向量化操作
- 大数据集未使用分块处理

**依赖问题**:
- pandas 版本过旧（0.23）
- numpy 存在兼容性问题

### 优先级行动：
1. 立即：更新数据科学库
2. 短期：重构核心处理函数
3. 中期：提高测试覆盖率到 80%
```

### 示例 3

**输入**:
```
加载我的整个项目文件夹:[上传移动应用项目]。审计代码质量、安全性和性能。
```

**输出**:
```
## 移动应用审计报告

### 安全问题:
- 硬编码的 API 密钥
- 未加密的本地存储
- 缺少证书锁定

### 性能瓶颈:
- 图片未压缩（平均 2MB/张）
- 网络请求未批处理
- 内存泄漏风险

### 技术债务:
- 弃用的 API 调用 15 处
- 硬编码字符串未国际化

### 修复优先级:
P0: 移除硬编码密钥，实施加密
P1: 优化资源，修复内存问题
P2: 更新弃用 API，支持国际化
```

## 🔧 参数调整建议

- **分析深度**: 可根据时间限制选择快速扫描（30 分钟）或深度审计（2-3 小时）
- **关注重点**: 可指定特定关注点（如安全优先、性能优先等）
- **项目规模**: 大型项目建议分模块审计
- **报告格式**: 可要求详细报告或执行摘要

## ⚠️ 注意事项

- 确保上传的文件不包含敏感信息（密码、密钥等）
- 大型项目可能需要分批处理
- 某些问题需要人工判断和上下文理解
- 审计结果应结合实际业务优先级
- 建议定期（每季度）进行审计
